## Introduction

You have written instructions on paper and arranged cards in order. Now you are going to do the same thing using blocks on a screen. Block programming is a way of writing programs by snapping pieces together instead of typing text. Each block is one instruction. When you join blocks in a sequence, you create a program. This is how many professional game designers start out, and it is the same logic that text-based programming uses, just shown in a different form.

## What We Will Learn

- What a block program is and how blocks represent instructions.
- How to assemble blocks into a sequence that moves a character.
- How to predict what a program will do before running it.
- How to identify and fix a block that causes the wrong result.

## Core Explanation

In block programming, each coloured block represents one instruction. You drag blocks onto a workspace and connect them in order from top to bottom, just like a written list of steps. When you press the run button, the program reads each block from the top and carries out its instruction.

The key blocks for movement are straightforward. A move block tells the character to take one step in the direction it is facing. A turn block rotates the character left or right. A wait block pauses before the next instruction runs. A sound block plays a short sound.

Reading a block program is the same skill as reading a written sequence. You start at the top, work through each block, and predict what will happen. This is called **tracing** a program. Tracing before running is one of the best habits a programmer can have, because it often reveals an error before the program even starts.

If the program does not produce the result you expected, you debug it. Look at the block where the result first went wrong, not the last block in the sequence. The error is usually earlier than it appears.

## Activity

**ScratchJr Pathfinder** (35 minutes)

If devices are available, use ScratchJr. If not, use the unplugged version described below.

**Device version:**

Step 1 (5 minutes): Open ScratchJr. Show students the move and turn blocks. Drag one move block onto the workspace and press run. Ask: "What happened? How far did the character move?" Drag a second move block below the first. Ask: "What do you predict will happen now?" Run it.

Step 2 (5 minutes): Together, build a program that moves the character to the right side of the screen. Start simply: three move blocks. Run it. Ask: "Did it go where we expected?"

Step 3 (15 minutes): Pairs work on their own challenge. They must move their character from the starting position to a small house or star placed at a specific spot on the screen. They must use at least one turn block. Before running, they read their program aloud and predict the path.

Step 4 (10 minutes): Each pair introduces one deliberate error into their partner's program by swapping one block. Partners must find the wrong block and explain what it caused.

**Unplugged version:**

Step 1 (5 minutes): Lay out a 4x4 grid. Place a character token at the bottom left. Place a star token at the top right. Show students the physical block cards: move forward, turn left, turn right.

Step 2 (5 minutes): Together, build a sequence of cards that would move the character to the star. Lay the cards in a vertical column. Read them aloud, moving the token after each card.

Step 3 (15 minutes): Pairs create their own path on the grid and build the matching card sequence. They swap grids with another pair and try to reproduce the other pair's path using their own card sequence.

Step 4 (10 minutes): One pair introduces a wrong card into the other's sequence. The other pair must identify it, say what went wrong, and swap it for the correct card.

## Real-World Connection

ScratchJr is used by children in over 150 countries. The programming ideas it teaches are identical to those used in professional tools. Move, turn, wait, and sound are the same concepts whether you are making a simple animation or writing code for a film studio. The blocks just make the structure visible.

## Check for Understanding

1. A program has these blocks in order: move forward, move forward, turn right, move forward. Draw the path the character takes on a small grid, starting at the bottom left and facing right.
2. You run a program and the character ends up in the wrong place. Which block should you check first: the last one, the first one, or the one just before the character went wrong?
3. What is the advantage of reading through a program before running it?

## Extension and Differentiation

**For students who need more support:** Work with a two-block program only. Move forward, then move forward again. Predict, run, and confirm. Add one block at a time, predicting each time before running.

**For students who want a challenge:** Write a program that visits three different locations on the grid in order, like a delivery route. The character must end up back at the starting position. How many blocks do you need?

## Summary

- Block programs are sequences of instructions shown as coloured shapes that snap together.
- Each block is one instruction. The program reads them from top to bottom.
- Tracing a program means reading it and predicting the result before running it.
- To debug, find the first block where the result went wrong, not the last block in the sequence.
